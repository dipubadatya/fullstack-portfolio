About This Project

When a patient urgently needs blood, clinics waste critical time calling through donor lists manually. There's no quick way to know which donors are nearby, available, and have the right blood group. This app puts that information on a map in seconds.

Two User Roles, Two Dashboards

Donors

  • Update profile and location coordinates
  • Toggle availability on or off
  • Control visibility in search results

Medical Staff

  • Search donors by blood group
  • Filter results within 5km or 10km radius
  • View donors on interactive map with contact details

Problems Solved

Building a location-based donor matching system requires handling several technical challenges. Here's what this project addresses.

Time-Critical Donor Search

Problem: Clinics manually call through donor lists during emergencies. This wastes critical time when every minute counts.

Solution: Blood group filtering and distance-based search display matching donors instantly. Map markers show exactly who's available nearby.

Distance-Based Filtering

Problem: A donor 50km away isn't useful in an emergency. Results need to be filtered by proximity to the clinic.

Solution: Radius filters (5km, 10km) calculate distance using coordinates. Only donors within range appear on the map.

Role-Based Access

Problem: Donors and medical staff need different interfaces. Showing search tools to donors or profile editing to staff creates confusion.

Solution: Role selection at registration. Protected routes redirect each role to their specific dashboard with relevant features only.

Donor Availability

Problem: Donors aren't always available. Showing unavailable donors in search results wastes the clinic's time.

Solution: Donors toggle their availability status. Only available donors appear in medical staff search results and on the map.

Secure Authentication

Problem: Donor information is sensitive. Unauthorized access could expose personal details and contact information.

Solution: Token-based authentication protects all routes. Sessions persist across page refreshes. Unauthorized users get redirected to login.

SPA Routing on Deploy

Problem: Single-page apps break on page refresh in production. Direct URL access returns 404 errors.

Solution: Vercel rewrite rules redirect all routes to index.html. React Router handles client-side navigation from there.

Technology Stack

Each tool was chosen for a specific purpose. Here's the complete stack and what each piece handles.

React 19

Frontend framework. Component-based UI with hooks for state management and side effects.

Vite

Build tool. Fast dev server with hot module replacement. Optimized production bundles.

React Router v7

Client-side routing. Handles navigation, protected routes, and role-based redirects.

Tailwind CSS

Utility-first CSS framework. Responsive design without writing custom stylesheets.

Leaflet

Mapping library with React Leaflet wrapper. OpenStreetMap tiles with dynamic marker rendering.

Framer Motion

Animation library. Smooth page transitions and component animations.

Axios

HTTP client. Handles API requests to the backend with interceptors for auth tokens.

Vercel + Render

Deployment platforms. Vercel hosts the frontend, Render hosts the backend API.

Authentication Flow

1

User submits login form with credentials

2

Backend validates and returns JWT token

3

Token stored in context and local storage

4

Protected routes check token before render

01 / Map Rendering

React Leaflet wraps the Leaflet library. OpenStreetMap tiles load on mount. Markers render based on filtered donor array from state.

02 / Distance Calculation

Haversine formula calculates distance between clinic and donor coordinates. Results filtered before passing to map component.

03 / State Management

React Context handles auth state globally. Component-level useState manages search filters, donor data, and UI state.
