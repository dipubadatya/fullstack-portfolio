About This Project

DIARY is a social platform built around writing. Users sign up, write posts, read what others have shared, and interact through likes, comments, and follows. The backend handles authentication, session management, and real-time updates. The frontend is server-rendered using EJS templates.

This isn't a static blog engine. It's a living, interactive platform where content flows between users in real time.

Problems Solved

Building a social platform requires handling multiple technical challenges. Here's what this project addresses.

User Authentication

Problem: Storing passwords in plain text exposes user data. Stateless apps struggle with persistent login states.

Solution: Passwords are hashed using Bcrypt before storage. Sessions are managed server-side with encrypted cookies. Protected routes verify sessions before granting access.

Real-Time Updates

Problem: Traditional HTTP requires page refresh to see new content. Users miss live interactions like new comments or likes.

Solution: Socket.io establishes persistent WebSocket connections. When someone likes a post or adds a comment, all connected users see it instantly without refreshing.

Image Storage

Problem: Storing images on the server filesystem doesn't scale. It increases server load and complicates deployments.

Solution: All image uploads (profile pictures, post attachments) go to Cloudinary. The database stores only URLs, keeping the server lightweight.

Email Workflows

Problem: Users forget passwords. Unverified accounts create spam. Manual email handling doesn't scale.

Solution: Nodemailer sends automated emails for account verification and password resets. Tokens expire after a set time for security.

Data Relationships

Problem: Social platforms have complex relationships — users follow users, comments belong to posts, posts belong to users.

Solution: MongoDB schemas use document references. Separate collections for Users, Posts, Comments, and Follows maintain clean relationships without data duplication.

Route Protection

Problem: Without proper checks, anyone can access private pages or manipulate other users' content.

Solution: Express middleware verifies session validity on every protected request. Users can only edit or delete their own posts.

Technology Stack

Each tool was chosen for a specific reason. Here's the complete stack and what each piece handles.

Node.js

Server runtime. Handles all backend logic, processes requests, and manages connections to external services.

Express.js

Web framework. Defines routes, handles middleware chains, manages sessions, and structures the application.

MongoDB Atlas

Cloud database. Stores users, posts, comments, and follow relationships. Mongoose provides schema validation.

EJS Templates

Server-side rendering. Generates HTML on the server with dynamic data before sending it to the browser.

Socket.io

WebSocket library. Pushes real-time updates to connected clients for likes, comments, and notifications.

Cloudinary

Image CDN. Handles all file uploads, image optimization, and delivery. Keeps the server stateless.

Nodemailer

Email service. Sends verification emails, password reset links, and transactional notifications.

Giphy API

GIF integration. Lets users search and embed GIFs directly into their posts and comments.

Under the Hood

Database Structure & Application Logic

Users

Account credentials, profile info, metadata

Posts

Written content, timestamps, author refs

Comments

Comment text, author ref, post ref

Follows

Follower-following relationships

01 / Data Models

Mongoose schemas with document references across User, Post, Comment, and Follow collections. No embedded documents — relationships stay clean and queryable.

02 / Auth Flow

User submits credentials → password compared against Bcrypt hash → session created with encrypted cookie → middleware checks session on protected routes.

03 / Project Structure

/models → Mongoose schemas
/routes → Express endpoints
/middleware → Auth checks
/views → EJS templates
/public → Static assets
